# DuckDB Schema Design — Alternative Data + Price Data

Generated by /office-hours on 2026-04-13
Mode: Builder
Status: DRAFT

## Problem Statement

Unify two distinct financial data layers in one DuckDB database so cross-layer
analysis is simple and fast:

- **Layer 1 — Prices:** Stocks, ETFs, crypto. Homogeneous OHLCV, daily, API-collected.
- **Layer 2 — Alternative/Macro:** Hundreds of indicators across freight, macro,
  sentiment, commodity, and industry categories. Heterogeneous, mixed frequency
  (daily / weekly / monthly / quarterly / 8x-per-year), manual Excel entry.

Core query targets:
- "Correlate weekly BDI changes with daily HMM (011200.KS) returns"
- "DRAM spot price vs SK Hynix (000660.KS) over last 2 years"
- "When OFR stress index > 2σ, what happens to KOSPI over next 20 days?"
- "Lead-lag: does CNN Fear & Greed predict forward SPY returns?"

## Premises

1. **Point-in-time correctness matters.** You plan to run regressions or backtests,
   so the schema must distinguish `observation_date` (what the value refers to) from
   `release_date` (when it was published). CPI for January 2025 gets released ~Feb 13.
   Using it on Jan 31 is look-ahead bias.
2. **Revisions matter.** GDP, nonfarm payrolls, and PMI get revised. First print ≠
   final value. Store revisions instead of overwriting.
3. **DuckDB is the right tool.** Columnar, fast joins, ASOF JOIN native, single-file,
   zero-ops, free. For single-user research at this data volume, it beats Postgres
   and ClickHouse on ergonomics.
4. **Manual Excel ingestion is a loader concern, not a schema concern.** The schema
   must not leak Excel-isms (no per-indicator columns, no merged cells, no frequency
   guessing). Loaders normalize into the long fact table.
5. **Prices stay wide, indicators go long.** Prices are homogeneous (always OHLCV);
   wide is natural. Indicators are heterogeneous; long is the only sane choice.

**Two fact tables by design.** Prices and indicators live in separate tables because
their shapes and ingestion cadences differ. They share a temporal axis (DATE), and
queries join them via `ASOF JOIN` on that axis. No need to force them into one table.

## Mixed-Frequency Alignment — Strategy Comparison

This is the central design decision. Three options:

### Option A — Long format at native frequency + ASOF JOIN at query time ✅ RECOMMENDED

Store each indicator at its real observation cadence. Never fabricate rows. At query
time, use DuckDB's `ASOF JOIN` to align lower-frequency data onto the price calendar.

**Pros:**
- Storage-efficient. Weekly indicators have ~52 rows/year, not 252.
- Zero fabricated data. You never "have" a value you didn't actually have.
- Point-in-time safe. ASOF on `release_date` trivially prevents look-ahead.
- Schema-stable. Adding a new indicator = INSERT into `dim_indicator`, no ALTER.
- Frequency changes handled automatically. If TrendForce switches a series from
  weekly to daily, you just start inserting more rows.

**Cons:**
- ASOF JOIN is unfamiliar to people coming from pandas. (But it's exactly what
  `pd.merge_asof` does.)
- You must remember that a "last known value" is not the same as the current value.
  Mitigated by clear view naming (`v_indicator_asof_*`).

### Option B — One wide daily table, forward-filled at load time ❌

Flatten everything to one row per business day with 200+ columns. Forward-fill weekly,
monthly, quarterly values across business days.

**Pros:** Trivial SELECTs. No JOIN needed.

**Cons (disqualifying):**
- **Destroys point-in-time correctness.** A forward-filled value on Feb 5 for "Jan CPI"
  is a lie — that value didn't exist until Feb 13. Every backtest becomes biased.
- **Schema churn.** Every new indicator = `ALTER TABLE ADD COLUMN`. Hundreds of indicators
  means a hundred migrations. Excel tabs don't update your DDL.
- **Sparse wide tables.** Most columns are NULL or forward-filled ghosts. Terrible for
  both storage and cognition.
- **Revisions break it.** Where does the revised GDP print go? Overwrite? Add a column?
  Both are bad.

### Option C — One fact table per frequency (`fact_daily`, `fact_weekly`, ...) ❌

Partition long format by frequency.

**Pros:** Feels organized.

**Cons:**
- **Frequency is metadata, not structure.** When TrendForce DRAM spot moves from
  weekly to daily, you have to migrate rows between tables.
- **N-way UNION** for any query that spans frequencies. Ugly.
- **TrendForce is already mixed inside one "category"** (daily spot + weekly contract +
  monthly contract). You'd split one logical source into three tables. Worse ergonomics
  than Option A with no compensating benefit.
- **No improvement over Option A** on performance. DuckDB's columnar scan already
  handles the "filter by indicator_id" case efficiently.

**Verdict: Option A.** Native frequency + ASOF JOIN is the only approach that
preserves correctness, scales with indicator count, and survives frequency changes.
Forward-filling is a *view concern*, not a *storage concern*.

## DDL — CREATE TABLE statements (DuckDB)

```sql
-- ============================================================
-- DIMENSIONS
-- ============================================================

CREATE TABLE IF NOT EXISTS dim_indicator (
    indicator_id        VARCHAR PRIMARY KEY,   -- 'BDI', 'US_CPI_YOY', 'TF_DRAM_DDR5_SPOT'
    name                VARCHAR NOT NULL,      -- 'Baltic Dry Index'
    category            VARCHAR NOT NULL,      -- 'freight' | 'macro' | 'sentiment' | 'commodity' | 'industry' | 'market'
    subcategory         VARCHAR,               -- 'shipping' | 'inflation' | 'memory' | ...
    country             VARCHAR,               -- 'US' | 'JP' | 'KR' | 'CN' | 'TW' | 'GLOBAL'
    frequency           VARCHAR NOT NULL,      -- 'daily' | 'weekly' | 'monthly' | 'quarterly' | 'fomc' | 'irregular'
    unit                VARCHAR,               -- 'index' | 'pct' | 'usd' | 'usd_per_ton' | 'k_contracts'
    source              VARCHAR NOT NULL,      -- 'Baltic Exchange' | 'BLS' | 'TrendForce' | 'CFM' | ...
    source_url          VARCHAR,
    release_lag_days    INTEGER,               -- typical days between obs_date and release_date (NULL = same-day)
    collection_method   VARCHAR NOT NULL,      -- 'manual_excel' | 'api_yahoo' | 'api_fred' | 'api_binance'
    description         VARCHAR,
    is_active           BOOLEAN DEFAULT TRUE,
    created_at          TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at          TIMESTAMP
);

CREATE TABLE IF NOT EXISTS dim_security (
    security_id         VARCHAR PRIMARY KEY,   -- '005930.KS', 'AAPL', 'BTC-USD', 'SPY'
    ticker              VARCHAR NOT NULL,      -- '005930'
    name                VARCHAR,               -- 'Samsung Electronics'
    asset_class         VARCHAR NOT NULL,      -- 'stock' | 'etf' | 'crypto' | 'index'
    exchange            VARCHAR,               -- 'KRX' | 'NASDAQ' | 'NYSE' | 'BINANCE'
    currency            VARCHAR NOT NULL,      -- 'KRW' | 'USD'
    country             VARCHAR,               -- 'KR' | 'US' | 'GLOBAL'
    sector              VARCHAR,               -- GICS sector
    industry            VARCHAR,               -- GICS industry
    index_membership    VARCHAR[],             -- ['KOSPI200', 'SP500']
    listed_date         DATE,
    delisted_date       DATE,
    is_active           BOOLEAN DEFAULT TRUE,
    created_at          TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- ============================================================
-- FACTS
-- ============================================================

CREATE TABLE IF NOT EXISTS fact_indicator_value (
    indicator_id        VARCHAR NOT NULL REFERENCES dim_indicator(indicator_id),
    observation_date    DATE    NOT NULL,      -- the date the value refers to
    release_date        DATE,                  -- when the value became public (NULL = unknown / same as obs)
    value               DOUBLE,                -- numeric value (primary storage)
    value_text          VARCHAR,               -- for non-numeric indicators (e.g. Fed policy text)
    revision            INTEGER NOT NULL DEFAULT 0,  -- 0 = first print, 1 = first revision, ...
    source_file         VARCHAR,               -- which Excel file / API call this came from
    ingested_at         TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (indicator_id, observation_date, revision)
);

CREATE TABLE IF NOT EXISTS fact_price (
    security_id         VARCHAR NOT NULL REFERENCES dim_security(security_id),
    date                DATE    NOT NULL,
    open                DOUBLE,
    high                DOUBLE,
    low                 DOUBLE,
    close               DOUBLE  NOT NULL,
    adj_close           DOUBLE,                -- dividend/split adjusted
    volume              BIGINT,
    source              VARCHAR,               -- 'yahoo' | 'naver' | 'binance' | 'pykrx'
    ingested_at         TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (security_id, date)
);

CREATE TABLE IF NOT EXISTS fact_fx (
    pair                VARCHAR NOT NULL,      -- 'USDKRW', 'USDJPY', 'USDCNY'
    date                DATE    NOT NULL,
    rate                DOUBLE  NOT NULL,      -- units of quote per 1 unit base
    source              VARCHAR,
    PRIMARY KEY (pair, date)
);

-- ============================================================
-- INDEXES (DuckDB auto-orders by PK; explicit indexes help point-lookups)
-- ============================================================

CREATE INDEX IF NOT EXISTS idx_ind_val_obs      ON fact_indicator_value(observation_date);
CREATE INDEX IF NOT EXISTS idx_ind_val_release  ON fact_indicator_value(release_date);
CREATE INDEX IF NOT EXISTS idx_price_date       ON fact_price(date);
CREATE INDEX IF NOT EXISTS idx_sec_asset_class  ON dim_security(asset_class);
CREATE INDEX IF NOT EXISTS idx_ind_category     ON dim_indicator(category);
```

### Why these choices

- **`indicator_id` as VARCHAR, not INTEGER surrogate.** You have ~200 indicators and
  you'll read `BDI` or `US_CPI_YOY` in every query. Human-readable IDs beat a surrogate
  join every time for research workloads. Storage cost is negligible in DuckDB.
- **`revision` in the PK.** This is how you store all vintages without losing any.
  `revision = 0` is the first print. To get "as known today" use `ORDER BY revision DESC
  LIMIT 1` or `MAX(revision)` — wrapped in a view.
- **`value` DOUBLE + `value_text` VARCHAR.** 99% of indicators are numeric. The text
  column is an escape hatch for things like FOMC statement tone labels, dot plot ranges,
  or CNN Fear & Greed categorical labels.
- **`source_file` on facts.** Auditability. When a value looks wrong, you need to know
  which Excel file it came from without digging through commit history.
- **`release_date` nullable.** Legacy Excel sheets often only have observation dates.
  Store NULL and backfill later. Queries that need point-in-time treat NULL as
  `observation_date + release_lag_days` via `COALESCE`.
- **No `dim_calendar`.** DuckDB's `generate_series` handles calendars on demand. Adding
  a calendar table is a Kimball reflex, not a DuckDB one.

## Analytical Views

```sql
-- ============================================================
-- v_indicator_latest — latest revision per (indicator, obs_date)
-- Use this for research when you want the "best known" value today.
-- NOT safe for backtests.
-- ============================================================
CREATE OR REPLACE VIEW v_indicator_latest AS
SELECT
    indicator_id,
    observation_date,
    release_date,
    value,
    value_text,
    revision
FROM fact_indicator_value
QUALIFY ROW_NUMBER() OVER (
    PARTITION BY indicator_id, observation_date
    ORDER BY revision DESC
) = 1;

-- ============================================================
-- v_indicator_pit — point-in-time view
-- For a backtest at date T, only values whose release_date <= T are visible.
-- Parametrize with a WHERE clause in your query.
-- ============================================================
CREATE OR REPLACE VIEW v_indicator_pit AS
SELECT
    iv.indicator_id,
    iv.observation_date,
    COALESCE(iv.release_date,
             iv.observation_date + INTERVAL (COALESCE(di.release_lag_days, 0)) DAY
    )::DATE AS effective_release_date,
    iv.value,
    iv.revision
FROM fact_indicator_value iv
JOIN dim_indicator di USING (indicator_id);
-- Usage: WHERE effective_release_date <= DATE '2024-06-30'

-- ============================================================
-- v_price_returns — daily simple + log returns per security
-- ============================================================
CREATE OR REPLACE VIEW v_price_returns AS
SELECT
    security_id,
    date,
    close,
    adj_close,
    adj_close / LAG(adj_close) OVER (PARTITION BY security_id ORDER BY date) - 1
        AS ret_1d,
    LN(adj_close / LAG(adj_close) OVER (PARTITION BY security_id ORDER BY date))
        AS log_ret_1d,
    adj_close / LAG(adj_close, 5) OVER (PARTITION BY security_id ORDER BY date) - 1
        AS ret_5d,
    adj_close / LAG(adj_close, 20) OVER (PARTITION BY security_id ORDER BY date) - 1
        AS ret_20d
FROM fact_price
WHERE adj_close IS NOT NULL;

-- ============================================================
-- v_indicator_daily_ff — indicators forward-filled onto business days
-- Convenience view for plotting and simple correlations.
-- WARNING: look-ahead unsafe. Do NOT use for backtesting without a PIT filter.
-- ============================================================
CREATE OR REPLACE VIEW v_indicator_daily_ff AS
WITH calendar AS (
    SELECT UNNEST(generate_series(
        DATE '2000-01-01',
        CURRENT_DATE,
        INTERVAL 1 DAY
    ))::DATE AS date
    -- filter to business days:
    WHERE EXTRACT(dow FROM UNNEST(generate_series(
        DATE '2000-01-01', CURRENT_DATE, INTERVAL 1 DAY))::DATE) NOT IN (0, 6)
),
latest AS (SELECT * FROM v_indicator_latest)
SELECT
    c.date,
    l.indicator_id,
    LAST_VALUE(l.value IGNORE NULLS) OVER (
        PARTITION BY l.indicator_id
        ORDER BY c.date
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS value_ff
FROM calendar c
LEFT JOIN latest l ON l.observation_date = c.date;

-- ============================================================
-- v_indicator_changes — pct and diff changes per indicator
-- Handles mixed frequency by computing change since previous observation.
-- "1-period change" means 1 week for weekly, 1 month for monthly, etc.
-- ============================================================
CREATE OR REPLACE VIEW v_indicator_changes AS
SELECT
    indicator_id,
    observation_date,
    value,
    value - LAG(value) OVER w  AS diff_1p,
    value / LAG(value) OVER w - 1        AS pct_1p,
    value / LAG(value, 4) OVER w - 1     AS pct_4p,
    value / LAG(value, 12) OVER w - 1    AS pct_12p
FROM v_indicator_latest
WINDOW w AS (PARTITION BY indicator_id ORDER BY observation_date);
```

## Example Queries

### 1. BDI weekly change vs HMM shipping stock daily returns

The canonical cross-layer query. Use `ASOF JOIN` to attach the latest-known BDI value
to each trading day.

```sql
WITH bdi AS (
    SELECT observation_date, value AS bdi, pct_1p AS bdi_wow
    FROM v_indicator_changes
    WHERE indicator_id = 'BDI'
),
hmm AS (
    SELECT date, ret_1d, ret_5d
    FROM v_price_returns
    WHERE security_id = '011200.KS'
)
SELECT
    hmm.date,
    hmm.ret_1d,
    hmm.ret_5d,
    bdi.bdi,
    bdi.bdi_wow
FROM hmm
ASOF LEFT JOIN bdi
     ON bdi.observation_date <= hmm.date
WHERE hmm.date >= DATE '2022-01-01'
ORDER BY hmm.date;
```

`ASOF LEFT JOIN` snaps each trading day to the most recent BDI observation on or
before that day. No forward-filling required, no fabricated rows.

### 2. DRAM spot price vs SK Hynix (000660.KS), 2-year overlay

```sql
SELECT
    p.date,
    p.adj_close                             AS hynix_close,
    p.adj_close / FIRST(p.adj_close) OVER (ORDER BY p.date) AS hynix_rebase,
    d.value                                 AS dram_spot,
    d.value / FIRST(d.value) OVER (ORDER BY p.date) AS dram_rebase
FROM fact_price p
ASOF LEFT JOIN (
    SELECT observation_date, value
    FROM v_indicator_latest
    WHERE indicator_id = 'CFM_DRAM_DDR5_SPOT'
) d ON d.observation_date <= p.date
WHERE p.security_id = '000660.KS'
  AND p.date >= CURRENT_DATE - INTERVAL 2 YEAR
ORDER BY p.date;
```

### 3. Point-in-time backtest: rolling correlation of CPI surprises and SPY

Only uses values that were actually public at each point in time.

```sql
WITH cpi_pit AS (
    SELECT effective_release_date AS known_on, value AS cpi_yoy
    FROM v_indicator_pit
    WHERE indicator_id = 'US_CPI_YOY'
),
spy AS (
    SELECT date, ret_20d
    FROM v_price_returns
    WHERE security_id = 'SPY'
)
SELECT
    spy.date,
    spy.ret_20d,
    cpi.cpi_yoy AS cpi_yoy_known_that_day
FROM spy
ASOF LEFT JOIN cpi_pit cpi
     ON cpi.known_on <= spy.date   -- PIT: only release dates <= trade date
WHERE spy.date BETWEEN DATE '2015-01-01' AND DATE '2024-12-31'
ORDER BY spy.date;
```

### 4. Regime detection — KOSPI returns when OFR stress is high

```sql
WITH stress AS (
    SELECT
        observation_date,
        value,
        AVG(value)    OVER w AS mu,
        STDDEV(value) OVER w AS sigma
    FROM v_indicator_latest
    WHERE indicator_id = 'OFR_FSI'
    WINDOW w AS (ORDER BY observation_date ROWS BETWEEN 251 PRECEDING AND CURRENT ROW)
),
stress_flag AS (
    SELECT
        observation_date,
        CASE
            WHEN (value - mu) / NULLIF(sigma, 0) > 2 THEN 'stress_high'
            WHEN (value - mu) / NULLIF(sigma, 0) < -1 THEN 'stress_low'
            ELSE 'normal'
        END AS regime
    FROM stress
),
kospi AS (
    SELECT date, ret_20d
    FROM v_price_returns
    WHERE security_id = 'KOSPI'
)
SELECT
    s.regime,
    COUNT(*)             AS n_days,
    AVG(k.ret_20d)       AS mean_fwd_20d,
    STDDEV(k.ret_20d)    AS vol_fwd_20d,
    AVG(k.ret_20d) / NULLIF(STDDEV(k.ret_20d), 0) AS sharpe_ish
FROM kospi k
ASOF LEFT JOIN stress_flag s
     ON s.observation_date <= k.date
GROUP BY s.regime
ORDER BY mean_fwd_20d DESC;
```

### 5. Lead-lag: CNN Fear & Greed → next-week SPY returns

```sql
WITH fg AS (
    SELECT observation_date, value AS fear_greed
    FROM v_indicator_latest
    WHERE indicator_id = 'CNN_FEAR_GREED'
),
spy_fwd AS (
    SELECT
        date,
        LEAD(adj_close, 5) OVER (ORDER BY date) / adj_close - 1 AS fwd_5d
    FROM fact_price
    WHERE security_id = 'SPY'
)
SELECT
    WIDTH_BUCKET(fg.fear_greed, 0, 100, 5) AS fg_bucket,
    COUNT(*)          AS n,
    AVG(s.fwd_5d)     AS mean_fwd_5d,
    STDDEV(s.fwd_5d)  AS vol_fwd_5d
FROM spy_fwd s
ASOF JOIN fg ON fg.observation_date <= s.date
WHERE s.fwd_5d IS NOT NULL
GROUP BY fg_bucket
ORDER BY fg_bucket;
```

### 6. Multi-indicator panel — one query, many signals, one stock

```sql
WITH base AS (
    SELECT date, adj_close, ret_5d
    FROM v_price_returns
    WHERE security_id = '005930.KS'   -- Samsung
)
SELECT
    b.date,
    b.adj_close,
    b.ret_5d,
    dram.value  AS dram_spot,
    nand.value  AS nand_spot,
    pmi.value   AS us_pmi,
    cn_pmi.value AS cn_pmi,
    fx.rate     AS usdkrw
FROM base b
ASOF LEFT JOIN (SELECT observation_date, value FROM v_indicator_latest WHERE indicator_id = 'CFM_DRAM_DDR5_SPOT') dram
    ON dram.observation_date <= b.date
ASOF LEFT JOIN (SELECT observation_date, value FROM v_indicator_latest WHERE indicator_id = 'CFM_NAND_SPOT') nand
    ON nand.observation_date <= b.date
ASOF LEFT JOIN (SELECT observation_date, value FROM v_indicator_latest WHERE indicator_id = 'US_PMI') pmi
    ON pmi.observation_date <= b.date
ASOF LEFT JOIN (SELECT observation_date, value FROM v_indicator_latest WHERE indicator_id = 'CN_PMI') cn_pmi
    ON cn_pmi.observation_date <= b.date
ASOF LEFT JOIN (SELECT date, rate FROM fact_fx WHERE pair = 'USDKRW') fx
    ON fx.date <= b.date
WHERE b.date >= DATE '2022-01-01'
ORDER BY b.date;
```

This is the query that shows why long format + ASOF JOIN wins. Adding a new indicator
to the panel is one more `ASOF LEFT JOIN` line. No schema change, no migration, no
wide-table refactor.

## Naming Conventions

**Tables:**
- `dim_*` for dimensions, `fact_*` for facts, `v_*` for views. Standard Kimball.

**`indicator_id` codes — UPPER_SNAKE with source/category prefix when disambiguation helps:**
- Pure indicators keep clean codes: `BDI`, `WCI`, `CCFI`, `OFR_FSI`
- Country-specific macro: `US_CPI_YOY`, `US_CPI_MOM`, `US_PMI`, `JP_CPI`, `KR_EXPORT`, `CN_PMI`, `TW_EXPORT`
- Policy rates: `FED_RATE`, `BOJ_RATE`
- TrendForce: `TF_DRAM_CONTRACT`, `TF_NAND_CONTRACT`, `TF_LCD_CONTRACT`, `TF_BATTERY`, `TF_PV`
- CFM: `CFM_DRAM_COMP`, `CFM_NAND_COMP`, `CFM_DRAM_DDR5_SPOT`, `CFM_SSD_SPOT`
- Sentiment: `AAII_BULL`, `AAII_BEAR`, `NAAIM`, `CNN_FEAR_GREED`, `SF_FED_SENT`
- Commodity: `COM_LITHIUM`, `COM_STEEL`, `COM_LUMBER`, `COM_GASOLINE`, `EIA_SUPPLY`

**`security_id`:**
- Equities with suffix: `005930.KS`, `000660.KS`, `AAPL.O`, `NVDA.O`
- Simple tickers for US majors: `SPY`, `QQQ` (leave `.N`/`.O` optional if unambiguous)
- Crypto with pair: `BTC-USD`, `ETH-USD`, `SOL-USD`

**Keep a discipline doc.** Add a `dim_indicator` insert for every new code before any
Excel sheet loads. The naming catalog becomes authoritative.

## Ingestion Notes (loader concerns, not schema)

- Each Excel file maps to one or more `indicator_id`s. The loader reads the sheet,
  parses the (date, value) columns, and UPSERTs into `fact_indicator_value`.
- For first-time loads, set `release_date = NULL` and let `v_indicator_pit` fall back
  to `observation_date + release_lag_days`. Backfill release dates when you have them.
- For revisions: ingest with `revision = MAX(revision)+1`. The PK prevents duplicates.
- API loaders for prices go directly into `fact_price`. Use DuckDB's `ON CONFLICT`
  to dedupe re-fetches.

## Open Questions

1. **Revisions for Excel-sourced data** — Do you actually track vintages, or is Excel
   a single source of truth you overwrite? If you overwrite, `revision` degenerates to
   always 0 and you lose PIT rigor. Decide before you load data.
2. **Corporate actions** — `adj_close` handles splits/dividends for backtest math, but
   do you also want a separate `fact_corp_action` table for analysis of event windows?
   Defer until you need it.
3. **FX coverage** — Do you need intraday FX or is daily enough? Daily covers 99% of
   cross-asset research.
4. **Storage target** — One `.duckdb` file in the repo, or a shared file on a drive?
   For solo research, one file in a `data/` folder that's `.gitignore`d is the move.

## Success Criteria

- `fact_indicator_value` loads all 200+ indicators from the Excel repo without schema
  changes.
- Any new indicator can be added by inserting one row into `dim_indicator`, no DDL.
- A cross-layer query like "BDI vs HMM" is one `ASOF JOIN` and runs in under 1 second
  on a full history.
- A point-in-time query is also one `ASOF JOIN`, just using
  `effective_release_date` instead of `observation_date`.
